## Stateless Loops Break in Production

A restart during a multi-step tool workflow can repeat actions (double charge, duplicate ticket, wrong update). Durable runtimes solve this by checkpointing every step and replaying safely.

<Mermaid
  title="Durable Agent Execution"
  chart={`
flowchart TD
  A[Start Step] --> B[Write Checkpoint]
  B --> C[Run Tool]
  C --> D[Persist Result]

  F --> G[Resume from Checkpoint]

## Runtime Contract

- Every step has deterministic input/output.
- Side-effecting actions are idempotent.
- Approval boundaries are explicit and replay-safe.
- Resume path is tested as a first-class scenario.

<RoleCallout role="dev">
Use idempotency keys for any external write. Persist keys before execution so retries don’t duplicate effects.
</RoleCallout>

<RoleCallout role="qa">
Create chaos tests: crash between checkpoint and tool return; verify recovery resumes exactly once.
</RoleCallout>

<RoleCallout role="pm">
Define which actions require approval and SLA expectations for paused tasks.
</RoleCallout>

<Callout type="warning" title="Production Gotcha">
“Retry on failure” without replay semantics is dangerous. Reliability requires state model design, not just retry middleware.
</Callout>
