## Why MCP Matters in Enterprise Systems

Most teams start with direct SDK calls to tools. It works for demos, then fails in production where you need permission boundaries, auditability, and standardized discovery.

Model Context Protocol (MCP) gives you a clean contract between model clients and tool servers so connectors are composable, testable, and governed.

## Design Principles

1. Tool servers are product interfaces, not utility scripts.
2. Every tool call must be attributable to actor, thread, and policy.
3. High-risk tools require explicit approval and output constraints.
4. Contract changes require semantic versioning.

<RoleCallout role="dev">
Treat each MCP server as a bounded context. Keep business invariants in the server, not in prompts.
</RoleCallout>

<RoleCallout role="qa">
Build contract tests that validate tool schema compatibility and policy behavior across versions.
</RoleCallout>

<RoleCallout role="pm">
Define tool risk tiers in product requirements. “Write access to customer records” is a product risk decision, not just an engineering one.
</RoleCallout>

<Callout type="warning" title="Production Gotcha">
If your tool descriptions are vague, model routing quality collapses. Spend as much effort on tool semantics as API implementation.
</Callout>
