## Why Final-Answer-Only Evals Fail

An agent can land on a good answer through unsafe steps. Enterprises need trajectory-level quality controls.

<Mermaid
  title="Trace-Aware Eval Pipeline"
  chart={`
flowchart TD
  A[Test Case] --> B[Run Agent Workflow]
  B --> C[Collect Trace]
  C --> D[Grade Outcome]
  C --> E[Grade Tool Sequence]
  C --> F[Grade Policy Compliance]
  D --> G[Release Decision]
  E --> G
  F --> G
`}
/>

## What to Grade

- Task correctness
- Tool selection correctness
- Forbidden action violations
- Latency and cost regression
- Human-approval boundary violations

<RoleCallout role="qa">
Treat eval datasets like product assets. Version them, annotate failure modes, and track drift over time.
</RoleCallout>

<RoleCallout role="dev">
Add eval hooks in runtime events (tool-call start/end, policy check, approval request) so traces are machine-gradeable.
</RoleCallout>

<RoleCallout role="pm">
Define go/no-go thresholds per use case. Not every feature needs 99% exactness, but every feature needs explicit release criteria.
</RoleCallout>

<Callout type="warning" title="Production Gotcha">
Teams often optimize what they can measure quickly (latency/cost) and ignore policy compliance. Grade both or quality will drift silently.
</Callout>
